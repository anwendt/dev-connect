# dev-connect Installation Guide

Status: Approved

## Scope

This guide describes the installation model implemented by the current repository. Published release commands must be generated from release artifacts and validated by the release pipeline.

Supported installation methods for the initial release:

- Binary installation.
- Helm installation.
- Kustomize installation.

The documented commands shall be generated from released artifacts and verified during release validation.

## Kubernetes Platform Prerequisites

- Kubernetes v1.30 or newer.
- Existing Rancher authentication model.
- Rancher-managed Kubernetes RBAC.
- Enterprise identity provider configured in Rancher where required.
- CNI with Kubernetes `NetworkPolicy` support.
- Private network reachability from gateway Pods to development servers on TCP port `22`.
- Platform logging backend, initially Azure Monitor / Log Analytics.

## Workstation Prerequisites

- Supported OS: Windows, Linux, or macOS.
- `kubectl`.
- Existing kubeconfig.
- VS Code Desktop.
- OpenSSH client.
- SSH authorization on the target development server.

## Kubernetes Installation Model

Installation is expected to be managed by platform GitOps.

Resources:

- Namespace.
- Deployment.
- Service.
- ConfigMap.
- NetworkPolicy.
- PodDisruptionBudget.
- optional HorizontalPodAutoscaler.
- RBAC.

Packaging:

- Helm chart.
- Kustomize base and overlays.

Generated manifests and Helm packages shall not be committed unless explicitly approved as examples.

Published Helm and Kustomize installation commands shall be derived from released chart packages and released Kustomize resources.

## Client Installation Model

The release pipeline builds cross-platform client binaries with:

- Windows amd64.
- Linux amd64.
- Linux arm64.
- macOS amd64.
- macOS arm64.

The binary shall not embed Kubernetes credentials or platform-specific secrets.

Local release binary generation:

```text
make build-all VERSION=vX.Y.Z
```

Generated binaries are written to `dist/` and are not committed.

Expected release binary names:

```text
dev-connect-vX.Y.Z-windows-amd64.exe
dev-connect-vX.Y.Z-linux-amd64
dev-connect-vX.Y.Z-linux-arm64
dev-connect-vX.Y.Z-darwin-amd64
dev-connect-vX.Y.Z-darwin-arm64
```

Published installation commands shall reference released binaries, checksums, SBOMs, and signatures.

## Helm Installation Model

Local validation:

```text
make helm-lint
make helm-template
make helm-package
```

Production deployments should consume the packaged chart generated by the release pipeline.

## Kustomize Installation Model

Local validation:

```text
make kustomize-build
```

The base deployment disables HPA by default. Autoscaling is available through the autoscaling overlay after environment-specific sizing data is approved.

## Release Artifacts

Release artifacts are generated by CI/CD:

- client binaries,
- gateway container image,
- Helm chart package,
- SBOM,
- signatures,
- build provenance,
- release notes.

## Production Readiness Gates

Before production rollout, the target environment must complete:

- Rancher RBAC validation for `kubectl port-forward`.
- Gateway Pod scheduling and readiness validation.
- NetworkPolicy validation with the target CNI.
- Azure Monitor / Log Analytics ingestion validation.
- Host key inventory onboarding through Platform GitOps.
- Windows, Linux, and macOS client smoke tests.

The formal production load test remains a separate activity and is intentionally not covered by this implementation slice.

Temporary CI artifacts are retained for 30 days. Release artifacts are retained permanently through the release and registry systems.

## Documentation Validation

User documentation shall be generated or validated from released artifacts wherever possible.

Documentation is not the authoritative source for implementation details. The implementation and released artifacts remain the single source of truth.
