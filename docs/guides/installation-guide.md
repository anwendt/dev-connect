# dev-connect Installation Guide

Status: Draft for Phase 10 review

## Scope

This guide describes the intended installation model. It does not provide final commands because implementation manifests, Helm charts, binaries, and workflows are not created in Phase 10.

Published installation commands shall only document installation methods that are generated and validated by the release pipeline.

Supported installation methods for the initial release:

- Binary installation.
- Helm installation.
- Kustomize installation.

The documented commands shall be generated from released artifacts and verified during release validation.

## Kubernetes Platform Prerequisites

- Kubernetes v1.30 or newer.
- Existing Rancher authentication model.
- Rancher-managed Kubernetes RBAC.
- Enterprise identity provider configured in Rancher where required.
- CNI with Kubernetes `NetworkPolicy` support.
- Private network reachability from gateway Pods to development servers on TCP port `22`.
- Platform logging backend, initially Azure Monitor / Log Analytics.

## Workstation Prerequisites

- Supported OS: Windows, Linux, or macOS.
- `kubectl`.
- Existing kubeconfig.
- VS Code Desktop.
- OpenSSH client.
- SSH authorization on the target development server.

## Kubernetes Installation Model

Installation is expected to be managed by platform GitOps.

Resources:

- Namespace.
- Deployment.
- Service.
- ConfigMap.
- NetworkPolicy.
- PodDisruptionBudget.
- optional HorizontalPodAutoscaler.
- RBAC.

Packaging:

- Helm chart.
- Kustomize base and overlays.

Generated manifests and Helm packages shall not be committed unless explicitly approved as examples.

Published Helm and Kustomize installation commands shall be derived from released chart packages and released Kustomize resources.

## Client Installation Model

The client will be distributed as cross-platform binaries:

- Windows amd64.
- Windows arm64 where supported.
- Linux amd64.
- Linux arm64.
- macOS amd64.
- macOS arm64.

The binary shall not embed Kubernetes credentials or platform-specific secrets.

Published binary installation commands shall be derived from released binaries and checksums.

## Release Artifacts

Release artifacts are generated by CI/CD:

- client binaries,
- gateway container image,
- Helm chart package,
- SBOM,
- signatures,
- release notes.

Temporary CI artifacts are retained for 30 days. Release artifacts are retained permanently through the release and registry systems.

## Documentation Validation

User documentation shall be generated or validated from released artifacts wherever possible.

Documentation is not the authoritative source for implementation details. The implementation and released artifacts remain the single source of truth.
